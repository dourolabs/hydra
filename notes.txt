## Kubernetes setup

### Prerequisites
- Docker must be installed and the daemon running
- `kind` and `kubectl` binaries installed and available in PATH
- A kind cluster named `local-dev` (create with the command below)

```
kind create cluster --name local-dev
```

To delete the cluster
```
kind delete cluster --name local-dev
```


## Building & Running

`cargo build` in the top level will build all projects.

Use `cargo check --workspace` for a quick compile pass before full builds.

`./scripts/docker-build.sh` will build docker images for codex and metis-server, and additionally push those images into kind's image repository.

`./scripts/service.sh` lets you start/stop/restart the server inside the kubernetes cluster. Typical usage is:

`./scripts/service.sh start` 
`./scripts/service.sh status` -- this will print out a URL that you can use with the metis cli to run commands against the server


`./scripts/docker-build.sh && ./scripts/service.sh restart` -- redeploy the current version of the code to the cluster. Note the URL from status above stays the same.

`./scripts/service.sh destroy` -- tear down the services in the cluster.


## Task system

Tasks are now plain Codex prompts.
The worker runs Codex with the provided prompt, using the repo context and environment variables supplied by the queue.

### Arguments

Tasks include a prompt string and a dictionary of `env` variables injected into the worker environment.

### Context



### Input/output

Task input/output is stored in files under the .metis directory.
Tasks will generate their output under .metis/output/.
The entire contents of the output directory will be copied to the corresponding input directory of child tasks.

Certain files within the output directory have special semantics that are understood by the metis system:
* last_message.txt -- contains the last message generated by the agent. This is 
* changes.patch -- contains the patch generated by the agent
* error.txt -- if this file exists, indicates that the task failed. The failure reason is the contents of the file.

The inputs from parent tasks to a task is stored under .metis/input/<parent name>/ .
The input directories contain the exact output from the parent task.

The setup for a task is as follows:
* unpack the context directory 
* create the .metis/input/<parent> directories and copy over the outputs from parent tasks
